# Проект E-Commerce на Django Oscar 4.1

Это документация по настройке и запуску проекта интернет-магазина на базе Django Oscar.
Проект использует Docker для изоляции окружения.

1. Стек технологий и архитектура
Проект представляет собой набор микросервисов, управляемых через Docker Compose. Каждый сервис отвечает за свою задачу и изолирован от других.

## 🛠️ Ключевые настройки и решения (HACKS)

В процессе настройки были решены архитектурные проблемы версии Oscar 4.1.

### 1. Конфигурация INSTALLED_APPS (в src/config/settings.py)
Для корректной загрузки всех модулей Oscar используется ручной список приложений.
Это решает проблему автозагрузки в данной версии.

### 2.  Первоначальная настройка и финальная очистка (Важный порядок действий)
В версии Oscar 4.1 существуют жесткие зависимости. Для первоначального запуска требуются «заглушки», но для стабильной работы их необходимо заменить на системные приложения и удалить.

Шаг 1: Создание папок-заглушек (ТОЛЬКО ДЛЯ ЗАПУСКА)
Когда делать: ПЕРЕД самым первым запуском (docker compose build / up), когда в INSTALLED_APPS еще нет системных приложений Oscar.

В папке src/ необходимо создать пустые папки-заглушки. Это временное решение для обхода ошибок зависимостей.
В папке src/ необходимо создать следующую структуру каталогов. Каждая папка является "заглушкой" (dummy app) для панели управления (Dashboard).

    └── apps.py             # (см. шаблон ниже)
└── communications_dashboard/
    ├── __init__.py
    └── apps.py
└── offers_dashboard/
    ├── __init__.py
    └── apps.py
... и так далее для всех списков ниже ...
Список всех необходимых папок-заглушек:

Шаблон файла apps.py для каждой папки:

Внутри каждой папки создайте файл apps.py с одинаковым содержимым. Просто замените имя класса и атрибут name.

Пример для src/catalogue_dashboard/apps.py:
from django.apps import AppConfig

class CatalogueDashboardConfig(AppConfig):

    name = 'catalogue_dashboard' 
    label = 'catalogue_dashboard' 
    # verbose_name можно задать для удобства, но не обязательно

Пример для src/orders_dashboard/apps.py:

class OrdersDashboardConfig(AppConfig):
    name = 'orders_dashboard'
    label = 'orders_dashboard'

Шаг 2: Подключение системных приложений и удаление заглушек
Когда делать: ПОСЛЕ того, как вы изменили INSTALLED_APPS, добавив в него полный список системных приложений Oscar (начиная с oscar.apps.analytics...).

Обязательное действие:Как только вы подключили системные приложения (из папки site-packages), все самодельные папки-заглушки из src/ должны быть немедленно удалены.

Почему это необходимо?Если оставить эти папки, возникнет ошибка Application labels aren't unique, duplicates: ..., так как Django обнаружит два приложения с одинаковым именем: ваше (в src/) и системное (внутри установленного пакета).

Что нужно сделать:

Удалить все перечисленные выше папки из каталога src/.
Убедиться, что строки типа 'catalogue_dashboard' удалены из списка INSTALLED_APPS.
Итог: Папки-заглушки — это временный «клей» для сборки проекта. После того как каркас собран, этот клей нужно убрать, чтобы он не мешал работе готового механизма.

### 3. Файл миграций `0008_assign_permissions_to_staff.py`
В приложении customer существует файл миграции 0008_assign_permissions_to_staff.py, который содержит жесткую зависимость от приложения analytics. Это архитектурный баг Oscar 4.1.

 Создание заглушки (НЕ СРАБОТАЛА)Изначально была предпринята попытка создать пустую папку-заглушку:src/oscar/apps/analytics/migrations/__init__.pyРезультат: Этот метод не решил проблему, так как система миграций Django все равно не смогла найти нужную структуру.

Решение: Мы не удаляем файл с локальной машины, а настраиваем контейнер так, чтобы он удалял этот файл при каждом запуске. Это гарантирует чистоту среды.
Важное замечание: Эта операция безопасна только при первом запуске на пустой базе данных.

Как это реализовано:

Где настроено: В файле docker-compose.yml.
Что настроено: Мы используем entrypoint для сервиса web. Эта команда выполняется первой при старте контейнера.
Команда удаления:
# Фрагмент из docker-compose.yml
services:
  web:
    # ... другие настройки ...
    entrypoint: ["/bin/sh", "-c", "rm -f /usr/local/lib/python3.11/site-packages/oscar/apps/customer/migrations/0008_assign_permissions_to_staff.py && exec gunicorn config.wsgi:application --bind 0.0.0.0:8000"]
rm -f ...: Команда, которая принудительно удаляет конфликтный файл внутри файловой системы контейнера.
&& exec gunicorn ...: После успешного удаления запускает сервер.
Результат: При каждом запуске (docker compose up) контейнер сначала удаляет "сломанную" миграцию, а затем стартует. Это позволяет обойти ошибку зависимостей, не изменяя исходный код самого пакета django-oscar.

### 4. Каталог для кастомизации: src/apps
Внутри папки src находится каталог apps. Это стандартное и правильное место для хранения ваших собственных, переопределенных приложений.

Что это за папки?

catalogue/
orders/
payments/
users/
Это пустые шаблоны. Они не содержат кода и не влияют на текущий запуск проекта. Их единственная цель — служить контейнером для вашего кода в будущем.

Зачем они нужны? (Принцип Overriding)Oscar — это фреймворк. Часто возникает необходимость изменить стандартное поведение. Например:

Добавить новое поле в модель User.
Изменить логику расчета стоимости доставки в корзине.
Написать свой обработчик платежей.
Правильный способ сделать это — «переопределить» (override) стандартное приложение. Вы берете код системного приложения (например, oscar.apps.customer.models) и копируете его в свою папку (src/apps/customer/models.py), а затем вносите изменения там. Django автоматически подхватит ваш код вместо системного.

## ⚙️ Как запустить проект

1.  **Установите Docker и Docker Compose.**
2.  **Скопируйте `.env.example` в `.env`** и заполните переменные (пароли).
3.  **Соберите образы:**
    ```bash
    docker compose build
    ```
4.  **Примените миграции и создайте суперпользователя:**
    ```bash
    docker compose run --rm web python manage.py migrate
    docker compose run --rm web python manage.py createsuperuser
    ```
5.  **Запустите контейнеры:**
    ```bash
    docker compose up -d
    ```
6.  **Откройте браузер:** http://localhost:8000/admin (для входа в админку).
